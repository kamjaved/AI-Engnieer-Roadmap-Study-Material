### First — what is LangMem, actually?

You just spent all of Lesson 5 hand-building a summarizer: count messages, decide when to compress, write the "fold old summary + new messages into one prompt" logic yourself, track a `covered_until_message_id` bookmark yourself. That was deliberate — you needed to feel the mechanics before trusting a library to hide them.

LangMem is the LangChain team's pre-built version of that exact job. Not a different idea — the same idea, professionally hardened. Think of it the way you'd think of writing your own debounce function in JS once, to understand it, and then reaching for lodash's `debounce` afterward because it handles edge cases you didn't think of (leading vs trailing calls, cancellation, etc).

Two things LangMem does better than your Lesson 5 version, right out of the box:

**It triggers on tokens, not message count.** Your `SUMMARIZE_TRIGGER_COUNT` counts messages — but messages are wildly different sizes. A "yes" is one message. A tool result dumping 40 sailings is also one message, but maybe 2000 tokens. Counting messages to decide "is this too big for the context window" is the beginner approach — it's easy to write, but it doesn't actually measure the thing you care about. Counting tokens is the production approach, because token budget is the actual constraint your model provider enforces.

**It tracks its own progress.** Instead of you maintaining a separate `covered_until_message_id` bookmark row, LangMem hands you back an object — `RunningSummary` — that remembers what it already compressed. You just have to keep passing that object back in next time.

### The shape of the API

```python
from langmem.short_term import summarize_messages, RunningSummary

result = summarize_messages(
    messages,                          # your current list of AnyMessage
    running_summary=running_summary,   # what LangMem gave you last time (or None)
    model=summarizer_model,
    max_tokens=...,
    max_tokens_before_summary=...,
    max_summary_tokens=...,
    token_counter=count_tokens_approximately,
)
```

It hands back a `SummarizationResult` with two things: `.messages` (a trimmed list ready to send to the LLM — old messages replaced by one summary message, recent ones kept raw) and `.running_summary` (a `RunningSummary` — `summary: str`, `summarized_message_ids: set[str]`, `last_summarized_message_id: str | None`). If nothing crossed the token threshold yet, `.running_summary` comes back `None` — nothing new happened, nothing to persist.

One thing worth flagging honestly: I checked this against LangMem's actual source and API reference, and the roadmap's 6.1 description lists `max_tokens_before_summary` and `max_summary_tokens` but leaves out `max_tokens` — which is actually a **required** parameter (no default), the hard ceiling on how big the returned message list is allowed to be. This is a small but real example of why you verify a library's real signature instead of trusting a doc summary of it — specs drift, source code doesn't lie. I'll include it below.

### The one gotcha that'll bite you if I don't call it out

LangMem tracks "what have I already summarized" using each message's own `.id` field — that's what `summarized_message_ids` is a set *of*. But look at Lesson 5's `get_context_for_turn()`: it builds `HumanMessage(content=msg.content)` — no `id` ever set. That was fine for Lesson 5, because Lesson 5's bookkeeping lived entirely outside the messages, in a separate `Summary.covered_until_message_id` column.

LangMem's bookkeeping lives *inside* the messages. No `.id` → LangMem has nothing stable to key its tracking on, and it'll effectively be blind between calls. So when we build LangChain messages for LangMem, we deliberately set `id=str(row.id)` — our own BigInteger PK, stringified. It's already unique and monotonically increasing, which is exactly the property LangMem wants. No new ID scheme required — and as a bonus, it means when LangMem hands back `last_summarized_message_id` as a string, we can just `int()` it straight back into our `covered_until_message_id: BigInteger` column, because it's a string we generated ourselves.

Variant A: LangMem's function-style summarizer (langmem.short_term.summarize_messages).

Same job as manual_summarizer.py's maybe_summarize(), but the trigger

from langchain_core.messages import AIMessage, AnyMessage, HumanMessage
from langchain_core.messages.utils import count_tokens_approximately

# mechanical difference from manual_summarizer.py's SUMMARIZE_TRIGGER_COUNT.
MAX_TOKENS = 3000                  # hard ceiling on the returned message list
MAX_TOKENS_BEFORE_SUMMARY = 3000   # crossing this triggers a summarization pass
MAX_SUMMARY_TOKENS = 512           # budget reserved for the summary text itself

# answer, so a cheap model at temperature=0 is the right call, not a
# frontier model.
_summarizer_model = ChatOpenAI(model="gpt-4.1-mini", temperature=0)

# Same role->class mapping as manual_summarizer.py — reused, not re-invented.
_ROLE_TO_MESSAGE_CLASS = {
    "user": HumanMessage,
    "assistant": AIMessage,
}


def _to_langchain_messages(rows: list[Message]) -> list[AnyMessage]:
    """
    Convert our DB rows into LangChain messages WITH `.id` set.

    This is the detail from the explanation above: LangMem's own progress
    tracking (RunningSummary.summarized_message_ids) is keyed off each
    message object's `.id`. Skip this and LangMem has nothing stable to
    track between calls — it'll look like every message is new, every time.
    """
    return [
        _ROLE_TO_MESSAGE_CLASS.get(row.role, HumanMessage)(content=row.content, id=str(row.id))
        for row in rows
    ]


async def summarize_with_langmem_function(
    db: AsyncSession,
    thread_id: str,
    message_rows: list[Message],
    running_summary: RunningSummary | None,
) -> RunningSummary | None:
    """
    Call LangMem's summarize_messages() and persist whatever it returns.

    `running_summary` is LangMem's bookkeeping object from the LAST call.
    On a fresh HTTP request there's no Python object sitting around to pass
    in — reconstructing one from the `summaries` table is exactly what 6.2
    builds next. Passing None here is legitimate: it just means "as far as
    LangMem knows, nothing has been summarized for this thread yet."
    """
    messages = _to_langchain_messages(message_rows)

    result = summarize_messages(
        messages,
        running_summary=running_summary,
        model=_summarizer_model,
        max_tokens=MAX_TOKENS,
        max_tokens_before_summary=MAX_TOKENS_BEFORE_SUMMARY,
        max_summary_tokens=MAX_SUMMARY_TOKENS,
        token_counter=count_tokens_approximately,
    )

    # Still under the token budget -> no summarization happened this call.
    # Nothing new to persist; hand back whatever we were given.
    if result.running_summary is None:
        return running_summary

    # last_summarized_message_id is a STRING — it's one of the ids we set
    # above (str(row.id)). Safe to cast back to int: we're the ones who
    # put it there in the first place.
    db.add(
        Summary(
            thread_id=thread_id,
            summary_text=result.running_summary.summary,
            covered_until_message_id=int(result.running_summary.last_summarized_message_id),
            strategy="langmem",
        )
    )
    await db.commit()

    return result.running_summary

### The problem, framed simply

`RunningSummary` is just a Python object sitting in memory. It's born inside one call to `summarize_messages()` and dies the moment that function returns — nothing keeps it alive between HTTP requests. FastAPI request #2 has zero memory of a variable that existed in request #1's stack frame. That's not a LangMem quirk, that's just... how processes work. Lesson 4 taught you this exact lesson already, just about a different object (the checkpointer's state) — anything that needs to survive between requests has to be a database row, not a Python variable.

So 6.1 persists two columns to `summaries` (`summary_text`, `covered_until_message_id`) every time LangMem hands back a fresh `RunningSummary`. 6.2's job is the reverse trip: on the *next* request, read those two columns back out of Postgres and rebuild a real `RunningSummary` object to hand to `summarize_messages()`.

### Why this isn't copy-paste

`RunningSummary` has three fields: `summary`, `last_summarized_message_id`, and `summarized_message_ids` — a whole **set** of ids, not just one boundary number. Our table only stores the boundary. So on the surface it looks like we're missing data.

Here's the resolution, and it's worth understanding *why* it's safe rather than just trusting it: I went and read LangMem's actual source for `summarize_messages()`. Turns out `last_summarized_message_id` is what does the real work — on every call, LangMem scans the incoming message list, finds the message whose `.id` matches it, and treats everything up to and including that position as "already summarized," everything after as "new." `summarized_message_ids` is a *safety net* — before summarizing a batch, it checks `if message.id in summarized_message_ids: raise ValueError(...)`, to catch a caller accidentally re-feeding it something already folded into the summary.

So to rebuild that set correctly, we need "every message id that's already been summarized so far." We don't have that stored explicitly — but we can derive it, because of a guarantee *our own schema* gives us that LangMem, as a general-purpose library, can't assume: `messages` is append-only, single-writer, auto-incrementing, never deleted (that was a deliberate Lesson 1 decision — no cascade deletes, remember). Given that guarantee, "every message with `id <= covered_until_message_id`" *is* exactly "every message summarized so far" — not an approximation, an exact reconstruction. A system that allowed deleting or reordering messages couldn't take this shortcut. That's precisely why LangMem's own dataclass stores the full set instead of trusting a single boundary — it can't assume your schema is this well-behaved. Ours is, so we get to exploit it.

One more thing worth catching now, not later: `summaries` is a shared table — Lesson 5's manual rows and this lesson's LangMem rows both land there, distinguished only by `strategy`. If this helper doesn't filter on `strategy == "langmem"`, it could pick up a `'manual'` row and hand LangMem a boundary computed by a completely different algorithm. Silent wrong answer, not a crash — the worst kind.

```python
# add to memory/langmem_summarizer.py

from sqlalchemy import select   # new import — everything else already in the file

async def load_running_summary(db: AsyncSession, thread_id: str) -> RunningSummary | None:
    """
    Rebuild LangMem's RunningSummary from what's persisted in `summaries`.

    Call this at the start of every /chat request, before
    summarize_with_langmem_function(), so LangMem knows what it already
    covered last time instead of starting from scratch every request.
    """
    latest = await db.scalar(
        select(Summary)
        .where(Summary.thread_id == thread_id, Summary.strategy == "langmem")
        .order_by(Summary.id.desc())
        .limit(1)
    )
    if latest is None:
        return None  # LangMem has never summarized this thread yet — nothing to rebuild

    # Exact reconstruction, not a guess — safe ONLY because `messages` is
    # append-only and never deleted (Lesson 1's schema decision). If this
    # app ever allowed deleting messages, this line would need to change.
    summarized_ids = await db.scalars(
        select(Message.id).where(
            Message.thread_id == thread_id,
            Message.id <= latest.covered_until_message_id,
        )
    )

    return RunningSummary(
        summary=latest.summary_text,
        summarized_message_ids={str(mid) for mid in summarized_ids},
        last_summarized_message_id=str(latest.covered_until_message_id),
    )
```

That means the caller must pass the **full thread history** into `summarize_with_langmem_function()`, not a pre-trimmed "only the new ones" slice the way Lesson 5's `maybe_summarize()` does. If the id it's looking for isn't in the list, the search just fails quietly and it treats everything as new again — no crash, just wasted tokens and a drifting summary. ### What `SummarizationNode` actually is

Here's the nice part: it's not a new algorithm. I went and read its source — `SummarizationNode` is a thin wrapper around the *exact same* `summarize_messages()` function you already called by hand in 6.1. Same trigger logic, same token counting, same `RunningSummary` object. The only thing it adds is packaging: it's shaped to drop straight into a LangGraph node slot, so instead of *you* calling the function and persisting the result to Postgres yourself (Variant A), the graph calls it automatically on every step, and stores its bookkeeping inside the graph's own state (Variant B).

Concretely, it reads messages from one state key (`messages`, by default), reads the running summary from a special state key called `context` (LangGraph's convention for node-local bookkeeping that still needs to survive across steps), and writes its trimmed output to a *different* key you name — that's `output_messages_key`. Our roadmap wants that key called `llm_input_messages`.

### Why the output goes to a separate key — this is the whole design

This is the detail the tracker's corrected note is about. `SummarizationNode` does **not** overwrite `state["messages"]`. It leaves it alone, completely untouched, growing forever. Instead it writes a second, trimmed list to `llm_input_messages` — and that's the list we actually hand to the LLM.

Think about why that split matters: `state["messages"]` is your permanent, full-fidelity record of the turn — the same role `messages` (the Postgres table) plays in your manual version, except this one lives inside the checkpoint. `llm_input_messages` is disposable — recomputed fresh, every single node run, from whatever the current running summary + recent messages happen to be. You never trust it to persist anything; you just trust it to be correct *this call*. Same "durable truth vs. what-gets-sent-this-turn" split you already internalized in Lesson 5 — LangMem just gives each half its own dedicated slot in graph state instead of you managing it by hand in Postgres.

### Wiring it into the graph — and one non-obvious edge

We add three nodes now instead of two: `summarize` → `agent` → (`tools` or `END`). Entry point is `summarize`, not `agent` — the roadmap's "added as a node before the agent node."

Here's the part that's easy to get wrong: when `tools` finishes running, where does it go? Lesson 3 wired `tools → agent` directly. For this variant, wire it `tools → summarize` instead — back through the summarizer, not straight to the agent. Reason: a tool call can dump a large result into the message list mid-turn. If a single user turn involves two or three tool calls back and forth, checking the token budget only once, at the very start, isn't enough — you want it re-checked after every loop iteration, not just once per HTTP request. Routing back through `summarize` is what makes that automatic.
