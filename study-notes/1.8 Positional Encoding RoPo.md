
## Positional Encoding Technique

Before we move on, we have to discuss the Positional Encoding technique. Let's ask why position matters.

### The Fundamental Flaw

We know how self-attention works by comparing every token to every other token simultaneously and in parallel. Since it reads everything at once, it has no sense of order. For example: "the dog bit a man" and "the man bit a dog". Both sentences are the same and mathematically identical to the model. Without knowing the position, AI can't understand grammar, syntax, and cause & effect.

We have to artificially inject position, or some information somehow, into the token before we feed it to the model. This technique is called Positional Encoding.

### Sinusoidal Encoding

This is the very first fix. In 2017, this technique was discussed in the research paper "Attention Is All You Need" published by Google. In this technique, researchers used a math trick based on sine and cosine waves. Every token is assigned a unique pattern of sine/cosine like a barcode. But the problem was it was an "absolute" and "static" encoding. If the model was trained on 4,000 tokens and you suddenly handed it a 5,000-token document, it crashed because it had no idea what the "barcode" was for the 4,001st token.

### RoPE (Rotary Positional Embedding)

To fix the limitation of sinusoidal encoding, researchers invented RoPE, which is now the industry standard (used in Llama, Mistral). Instead of adding a static barcode to the word, RoPE rotates the word vector in mathematical space. Think of a token vector like the hand of a clock. If a word is in position 1, and the vector is rotated 4 steps, it will be on position 5. This is a brilliant technique because it automatically calculates relative distance. If the model wants to know the relationship between Token 5 & 8, it just looks at the angle/step distance (e.g., 3). The same applies for token 100 & 103. Because it relies on relative angles, it allows modern models better extrapolation and better long-context behavior.

## Flash Attention: Hardware (GPU) Bottleneck

To understand Flash Attention, first we need to understand GPU hardware. The hardware hierarchy inside a GPU chip (from fastest to slowest) is:

* **Registers:** Computation happens here (Super fast).


* **SRAM / Shared Memory:** Very fast.


* **L2 Cache**.


* **GPU RAM (HBM):** Data lives here (Much slower).

```
    GPU Chip

    ├── Registers
    │    Super Super Fast (Computation happen here)
    │
    ├── Shared Memory (SRAM)
    │    Very Fast
    │
    ├── L2 Cache
    │    Fast
    │
    └── HBM (GPU RAM)
        Much Slower (Data lives here)
```


Modern GPUs like the NVIDIA A100 or H100 are incredibly efficient and fast; they can execute trillions of floating-point operations every second. The problem occurs because although the GPU is capable of performing millions of operations, the memory only supplies too little compared to the GPU's capabilities. For example, if the GPU can calculate 1,000 numbers/sec but the memory (RAM) only supplies 100 numbers/sec, as a result, the GPU waits 90% of the time. It's like you have an F1 car which can easily reach 350km/h, but you are driving in city traffic.

That is where researchers asked: why are we storing the entire **Attention Matrix** at all? 
> **Attention Matrix:** After the model create vectors (Q, K, V), it calculates and makes the Attention Matrix from (Q & K) and stores it in HBM.

This is where **Flash Attention** comes into the picture. It breaks the Attention Matrix into small blocks which are called **_Tiles_**. It feeds and sends these to the fast SRAM at once, and then moves to the next block/tile. This method cuts the memory usage (from O(N²) down to O(N)) and makes it 2x to 4x faster. (Note: Don't confuse this with KV cache; both solve different problems.)

Flash Attention is the single software breakthrough that made a 128K context window size physically possible.

## Lost in the Middle Problem

It's an important topic in GenAI development. Let's clear a confusion first: when we say a model supports a 128K or 1 million token context window, the model doesn't guarantee that it pays equal attention to every part of the input. They mostly remember the beginning and the end. This phenomenon is known as the "Lost in the Middle" problem.

**Example:** If you provide a 50-page doc to the model and put the most important fact either on page 1 or page 50, the model will find it with 95% accuracy. But if you bury the fact on page 25, the model might miss it and give you a wrong answer. This happens because Attention becomes diluted.

### How to Reduce This Problem

AI researchers are well aware of this limitation.

* In Retrieval-Augmented Generation (RAG) they rely on "just stuffing everything" in the context

* 2nd way is to re-rank chunks and place the highest relevance documents at the very beginning or the very end.


* Another way is context compression, where the system creates a summary of the conversation.
