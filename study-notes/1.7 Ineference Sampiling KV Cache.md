## 1. The Core Mechanic: Logits and Probabilities

When a user sends a prompt to an LLM, the model processes it and calculates a massive list of scores for every single word in its vocabulary to decide what word should come next. These raw mathematical scores are called **Logits**.

Before the model can pick a word, it passes these logits through a mathematical function (called *Softmax*) to convert them into **Probabilities** that add up to 100%.

For example, if the prompt is *"The boiling point of water is 100 degrees..."*

* `Celsius` = 90%
* `Fahrenheit` = 8%
* `banana` = 0.001%

The model now has to make a choice. *How* it makes that choice depends entirely on the sampling parameters you set.

## 2. Choosing the Next Word: Decoding & Sampling Strategies

Here is a breakdown of the specific strategies and dials used to pull a final token from that probability list.

### Greedy Decoding 🔴 (The Deterministic Baseline)

* **What it is:** The simplest method possible. The model looks at the probability list and **always picks the #1 highest probability token**. Period.
* **The Result:** The output is completely deterministic. If you run the exact same prompt 10 times, you will get the exact same answer 10 times.
* **The Problem:** It often leads to highly repetitive, dry, and looping text. It is almost never used for chatbots, but it is excellent for strict factual tasks like data extraction or simple translation.

### Temperature 🔴 (The Creativity Dial)

* **What it is:** Temperature is a dial that scales the logits *before* they are converted into percentages.
* **How it works:**
* **Temperature < 1.0 (e.g., 0.2):** It makes the high-probability tokens even higher and crushes the low-probability tokens. The model becomes highly focused, conservative, and robotic. *(Use for: Coding, Math, Legal summaries).*
* **Temperature > 1.0 (e.g., 1.5):** It flattens the distribution. The #1 word and the #15 word suddenly have very similar probabilities. The model starts taking risks and becomes highly creative (or hallucinates). *(Use for: Brainstorming, Poetry).*

 Example:

 ```
          Raw logits: [0.1, 2.5, 0.3, 1.2]

          temperature = 1.0 (default):
          scaled = logits / 1.0 = [0.1, 2.5, 0.3, 1.2]
          probabilities = softmax(scaled) = [0.05, 0.88, 0.03, 0.04]
                        ↑ High concentration on highest probability

          temperature = 0.5 (cold):
          scaled = logits / 0.5 = [0.2, 5.0, 0.6, 2.4]  ← Amplified differences
          probabilities = softmax(scaled) = [0.002, 0.997, 0.0006, 0.0006]
                        ↑ EVEN MORE concentrated on highest probability

          temperature = 2.0 (hot):
          scaled = logits / 2.0 = [0.05, 1.25, 0.15, 0.6]  ← Flattened differences
          probabilities = softmax(scaled) = [0.15, 0.35, 0.18, 0.32]
                        ↑ More evenly distributed, higher randomness
```
### Top-k 🟡 (The Hard Cutoff)

* **What it is:** You set a fixed number, like $k = 5$.
* **How it works:** The model looks at its full vocabulary, ranks every token by probability, keeps only the top $k$ tokens, and samples from that smaller pool.
* **Why it matters:** It is simple and easy to control, but the cutoff is rigid. Even when one token is obviously the best choice, Top-k still keeps the same number of alternatives.

**Example**

Suppose the prompt is: **"The capital of France is"**

The model assigns probabilities like this:

| Token | Probability |
| --- | ---: |
| Paris | 40% |
| Lyon | 25% |
| Marseille | 15% |
| London | 8% |
| Berlin | 5% |
| Apple | 2% |
| Cat | 2% |
| Banana | 1% |
| Quantum | 1% |
| ... | ... |

If you set $k = 3$, the model keeps only the top 3 tokens:

| Token | Probability |
| --- | ---: |
| Paris | 40% |
| Lyon | 25% |
| Marseille | 15% |

Everything else is ignored. The remaining probabilities are then normalized:

* $40 + 25 + 15 = 80$
* Paris = $40 / 80 = 50\%$
* Lyon = $25 / 80 = 31.25\%$
* Marseille = $15 / 80 = 18.75\%$

The model then randomly picks from those three tokens.

* **The Problem:** It is a rigid, fixed cutoff. If a model is 99% sure the next word is "Celsius," Top-k still forces it to keep many irrelevant words in the candidate pool, which can lead to awkward or weird word choices.

### Top-p / Nucleus Sampling 🔴 (The Dynamic Cutoff)

* **What it is:** Instead of a fixed *number* of words, you set a target *cumulative probability*, like $p=0.90$ (90%).
* **How it works:** The model goes down the ranked list and adds up the probabilities of the words until it hits 90%.
* If the model is highly confident, it might hit 90% after just 2 words (e.g., `Paris` [80%] + `Berlin` [10%]). It deletes all other words.
* If the model is confused, it might take 100 words to add up to 90%.


* **Why it matters:** Top-p is **dynamic**. It adapts to how confident the model is at any given step. This is the industry standard for modern LLMs and is generally preferred over Top-k.

> **Production Rule:** Never tweak Temperature and Top-p at the same time. If you want to adjust creativity, leave Top-p at 1.0 and change the Temperature.

To help visualize exactly how these three dials manipulate the model's choices in real-time, use the interactive simulator below.

### Beam Search 🟢 (The Look-Ahead Strategy)

* **What it is:** Instead of just guessing the next word, Beam Search explores multiple different paths simultaneously (e.g., keeping 3 different possible sentences going at once) and eventually picks the path that has the highest overall probability for the *entire sequence*.
* **Current Status:** It is excellent for strict translation tasks, but it is heavily deprioritized in modern chat LLMs because it requires too much compute memory and often results in highly generic text.

## 3. Engineering Performance: Memory and Reproducibility

### The Seed Parameter 🟡

* **What it is:** A specific number you pass into the API (e.g., `seed=42`).
* **Why it matters:** Because sampling (like Top-p) involves randomness, an LLM will give you a slightly different answer every time. In production, if you are running automated software tests or debugging a failure, you need the model to be predictable. Setting the Seed parameter forces the underlying random number generator to output the exact same sequence, ensuring **reproducibility** for debugging.

### KV-Cache 🔴 (The Speed & Memory Bottleneck)

This is a critical architectural concept for anyone deploying models in production.

* **The Problem:** In a Transformer, every time a new token is generated, the model has to use the Attention mechanism to look back at *every single previous token* in the prompt to establish context. If you have a 10,000-word prompt, re-calculating the mathematical vectors (Keys and Values) for all 10,000 words *for every single new word generated* would be impossibly slow. ($O(n^2)$ time complexity).
* **The Solution:** The **KV-Cache** (Key-Value Cache). Once the model calculates the Key and Value vectors for a word, it saves them in a high-speed memory cache on the GPU.
* **The Tradeoff:** The KV-Cache reduces the computation time from $O(n^2)$ to $O(n)$, making inference incredibly fast. However, **it consumes a massive amount of VRAM**. As context windows grow to 128k or 1 Million tokens, the KV-Cache grows linearly. A 128k context on a 7B parameter model can take up over 60GB of memory just for the cache alone, becoming the primary bottleneck for serving long-context requests.

## Unlocking the Interview Questions 🔒

### Beginner Questions

**Q: What is temperature and how does it affect LLM outputs?**
Temperature is a scaling factor applied to raw logits before they are converted into probabilities. A low temperature (<1.0) sharpens the distribution, making the model highly focused, predictable, and conservative (best for facts and code). A high temperature (>1.0) flattens the distribution, giving lower-probability words a higher chance of being picked, resulting in more diverse, creative, or unpredictable text.

**Q: What is the KV-cache and why is it important?**
The KV-cache stores the previously computed Key and Value vectors for tokens in a sequence so the model doesn't have to recompute them from scratch for every new word it generates. It is essential because it drastically reduces the computational latency of text generation, though it requires significant GPU memory to maintain.

**Q: When would you use greedy decoding vs nucleus sampling (Top-p)?**
You use greedy decoding (always picking the #1 token) when you need strict deterministic outputs where accuracy is paramount, such as data extraction to JSON or simple translation. You use nucleus sampling (Top-p) for chat, creative writing, or conversational agents, as it allows for natural word variation while still safely cutting off completely illogical word choices.

### Intermediate Questions

**Q: Explain the difference between temperature, top-p, and top-k. When would you use each?**

* **Temperature** alters the underlying probability distribution itself, changing the mathematical weight of the options. Use it to control overall creativity.
* **Top-k** is a hard filter that only allows the top *X number* of tokens to be considered, regardless of their actual probability. It is older and rarely used alone now.
* **Top-p** is a dynamic filter that cuts off the list based on *cumulative probability*, adapting to how confident the model is. Use it as the standard mechanism to prevent the model from picking nonsensical words during generation.

**Q: Why does KV-cache matter for serving long-context requests?**
Because the memory footprint of the KV-cache scales linearly with the sequence length. If a user uploads a massive PDF (long context window), the KV-cache for that single request can consume tens of gigabytes of GPU VRAM. It matters because it becomes the absolute hard limit on how many concurrent users a single GPU server can handle before running out of memory.

**Q: Why would you set a seed parameter when calling an LLM API?**
To guarantee deterministic, reproducible outputs during inference. This is strictly necessary for automated CI/CD testing, debugging unexpected AI behaviors, or evaluating prompt variations, as it removes the randomness inherent in standard sampling methods.

### **Cheat Sheet: Lesson 1.7**

| Parameter | What it Does | Production Use Case |
| --- | --- | --- |
| **Greedy Decoding** | Always picks #1 token. | Strict data extraction, JSON generation. |
| **Temperature** | Alters probability math (Low = strict, High = creative). | The primary dial for adjusting tone and logic. |
| **Top-p (Nucleus)** | Dynamic cutoff based on total probability. | Standard chat generation; prevents hallucinations. |
| **Top-k** | Fixed cutoff based on a set number of words. | Legacy control; mostly replaced by Top-p. |
| **KV-Cache** | Saves past token math in GPU memory. | Speeds up inference but hogs massive VRAM at long contexts. |
| **Seed** | Locks the random number generator. | Software testing and debugging. |
